# Ecommerce Reranker

A from-scratch recommendation and reranking system built on behavioral event data.
Implements Matrix Factorization via Alternating Least Squares (ALS) to learn user
and item embeddings, then uses dot product scoring to rerank items per session.
Evaluated with NDCG@K against position and random baselines.

The pipeline is fully automated with Apache Airflow — running monthly to ingest
new data, retrain the model, and evaluate improvement over time.

## How It Works

### The Core Problem
An ecommerce platform shows users items in a ranked list. The default ranking
is biased — items shown at position 1 get clicked more simply because they appear
first, not because they are better. This is called **position bias**.

The goal is to learn true item quality and user preference from behavioral signals
(clicks, cart adds, purchases) and produce a ranking that corrects for this bias.

### The Approach

```
Raw events (clicks, purchases)
        ↓
Feature engineering (DuckDB SQL)
        ↓
Sparse user-item interaction matrix
        ↓
ALS Matrix Factorization
        ↓
User embeddings (N × 32) + Item embeddings (M × 32)
        ↓
Dot product score per (user, item) pair
        ↓
Reranked results + NDCG@K evaluation
```

### ALS in One Paragraph
ALS factorizes the interaction matrix **R** (users × items) into two smaller
matrices **U** (users × K) and **I** (items × K) such that `U × Iᵀ ≈ R`.
Each user and item is represented as a K-dimensional vector. The dot product
of a user vector and item vector predicts how much that user would like that item.
ALS alternates between solving for **U** (fixing **I**) and solving for **I**
(fixing **U**) — each solve has a closed-form solution, making training fast
and stable. Based on Hu, Koren & Volinsky (2008).

## Quickstart

### Prerequisites
```bash
pip install pandas numpy scipy pyarrow duckdb scikit-learn
```

### Run the full pipeline manually
```bash
python scripts/01_generate_synthetic_data.py
python scripts/02_feature_engineering.py
python scripts/03a_interaction_matrix.py
python scripts/03b_als_model.py
python scripts/03c_rerank.py
```

### Config
Copy the example config and fill in your values:
```bash
cp pipeline.conf.yaml.example pipeline.conf.yaml
```

Key parameters in `pipeline.conf.yaml`:
```yaml
model:
  als:
    k: 32          # embedding dimension
    n_iters: 15    # training iterations
    lambda_reg: 1.0  # regularization strength
    alpha: 40        # confidence scaling (Hu et al.)

data:
  n_sessions: 50000  # sessions per pipeline run
```

## Automated Pipeline (Airflow)

The pipeline runs automatically on the 1st of every month via Apache Airflow.
Each run appends new behavioral data and retrains the model on the full history —
the model improves as more data accumulates.

```
Run 1: 50,000 sessions  →  total:  50,000  events
Run 2: 50,000 sessions  →  total: 100,000  events
Run 3: 50,000 sessions  →  total: 150,000  events
```

### Setup
1. Add `RERANKER_PATH` to your Airflow `.env`:
   ```
   RERANKER_PATH=/absolute/path/to/your/reranker
   ```

2. Add Docker socket mount and provider to `docker-compose.yml`:
   ```yaml
   volumes:
     - //var/run/docker.sock:/var/run/docker.sock
     - <your-reranker-path>:/opt/reranker
   _PIP_ADDITIONAL_REQUIREMENTS: apache-airflow-providers-docker python-dotenv pyyaml
   ```

3. Copy DAG file to your Airflow dags folder:
   ```
   reranker_pipeline.py → <your-airflow-docker-dir>/dags/
   ```

4. Restart Airflow, unpause the DAG, and trigger manually to verify.

### Task Flow
```
[generate_data] → [feature_engineering] → [build_matrix] → [train_als] → [rerank_evaluate]
    ~90s                ~30s                   ~10s             ~15s            ~20s

Total runtime: ~3 minutes
```

## Key Design Decisions

**Why ALS over LightGBM LambdaRank?**
ALS is implemented from scratch in numpy — the goal was to understand the
internals of collaborative filtering rather than call a library. The embeddings
it produces can also feed into a downstream LTR model as features, which is
a natural next step.

**Why DuckDB for feature engineering?**
DuckDB runs in-process with Python, needs zero setup, reads Parquet natively,
and supports the same analytical SQL (window functions, CTEs) as BigQuery.
The feature engineering SQL written here is directly portable to BigQuery
for a production migration.

**Why synthetic data?**
Public ecommerce datasets don't expose ground-truth relevance signals needed
to properly evaluate a reranker. Generating data allows full control over
signal strength, position bias level, and behavioral patterns — and lets
us verify the model is learning the right things.

**Why monthly retraining?**
Balances model freshness against compute cost. Each run appends a new month
of behavioral data and retrains on the full history — the model naturally
improves as the interaction matrix becomes denser.

## Known Limitations

**Residual position bias**
ALS scores are still mildly correlated with original shown position. True
debiasing requires Inverse Propensity Scoring (IPS) during matrix construction
— a natural next improvement.

**No query-aware embeddings**
ALS produces user and item vectors but has no concept of a search query.
The search reranking improvement comes from user-item affinity, not
query-item relevance. Adding a query tower (Two-Tower model) would
significantly improve search surface performance.

**Synthetic data ceiling**
The high NDCG scores (0.82+) reflect that the model is learning patterns
from data it was also evaluated on. A proper offline evaluation would use
a time-based holdout set.
